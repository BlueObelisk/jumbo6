File formats

Here Iâ€™m commenting on the very different approach that CML takes to traditional chemical information which you can see on Chemical File Format on Wikipedia. (I shall rely increasingly on WP to explain things). In this youâ€™ll see that there are tens if not hundreds of file formats.

Program formats

These normally arise because a software developer needs a format to export and import information from their system. The format is designed to hold the information necessary to support the state of the system so that it can be re-read at a later stage. Typical systems that do this are compoutational chemistry programs (where more computational cycles may be required) and chemical editors or modelling programs. In general these formats are program-specific (sometimes manufacturer-specific). There have been times when the formats were seen as trade secrets and, in general manufacturers have not worked hard towards interoperability - where program A can read the output of program B from a different manufacturer.

Because of the effort to write software most authors see their system as the local centre of their universe. Design for interoperability has been low. There is no easy way that (say) the output of GAMESS can be directly read into Gaussian or vice. This has led either to either:

  • a series of Foo2Bar converters - specialist codes to link two large systems. The problem here is that there have to be N(N-1)/2 converters for N programs, and also that it is boring, especially to maintain so that such converters are incomplete and decay.
  • generic converters (especially from the BlueObelisk interoperabilty movement) such as OpenBabel or libraries such as CDK, JOELib and JUMBO.

CML has been heaviliy informed by these exercises. We take the approach: â€œHere is an important program; how much of the output can CML capture? and what is the information loss?â€? So CML attempts to address much of the information described on the File Formats page.

In practice there is a lot not captured as many programs expose parts of their internal data structure but without clear semantics. This is compounded by shorthand poorly documented fields, especially in the input of programs. Complete interoperability between existing programs is probably impossible, but the next generation of CML-aware programs will help to tackle this.

Data formats

Several commendable communal efforts have created data formats which are independent of the software that is expected to produce of process them. Historically the two most important areas were crystal structures (CIF and PDB) and spectroscopy (JCAMP). The latter is now being addressed by the AnIML group. These are developed to

  • aid interoperability between different suppliers (e.g. a JCAMP output by manufacturer-A should be readable by manufacturer B
  • add semantic markup - making it clear exactly what a field means
  • act as a means for archival (very strong in crystallography)
  • act as a tool for authors to deposit their experimental information at time of publication or elsewise.

petermr has been part of the CIF effort for many years and the CIF process and structure has heavily informed the developed of CML. CIF and JCAMP were developed before XML and - in an ideal world - will benefit from conversion, but it is extremely difficult to convert formats in a large community. Thus although mmCIF is a more extensible format, many protein crystallographers still use PDB.
Documents

The other major motivation for CML is to create semantically rich living publications. The current method of publication uses a paper-like layout which would not have seemed strange to Kekule and Beilstein 150 years ago. Yet the rest of the world is adopting Google-based mashups, instant online information, textable videos, etc. In chemistry if we want to know the height of a peak in a spectrum we have to print the paper and measure it with a ruler. Despite the fact that it used to be digital when it was measured

So CML is designed to help create compound documenst which include chemistry. A typical chemistry paper, as I have blogged, contains text, references, chemistry, maths and diagrams. All of these are expressible in XML and Iâ€™m hoping to demonstrate some of this at the ACS meeting in a weekâ€™s time.
Compound documents

In general a document including CML may have chunks of XHTML, MathML and SVG. Therefore we have to distinguish between them by namespaces. Weâ€™ll show this in a subsequent post, but we note here that in general all CML documents should include the namespace:

http://www.xml-cml.org/schema

This is a unique name on the planet and serves to indicate the material is CML.

There is no required format for a CML document, but it may turn out useful to start it with <cml>. However itâ€™s perfectly allowed to start it with <molecule>, <spectrum> or any other CML element. Thus the smallest CML document might look like:

This is the â€œhello worldâ€? of CML. It says â€œI am a CML documentâ€?. Hereâ€™s a more chemical one:

Welcome to CML - September 1st, 2006

Welcome to the CML blog! This is the first posting and will cover some general principles and procedures. We hope you will respond with comments - but they will be public! In the medium term we hope that the blog can concentrate and formalise much of the usage and semantics of CML. The blog does not supersede the cml-discuss list, but may be more spam free and should allow better navigation of topics.
This blog is offered to anyone interested in CML and more generally aspects of XML for physical sciences. CML is an information design which covers a large area of general chemistry and has several complementary aspects:

  • formalisation of the semantics of chemistry. Before XML there has not been a simple way of doing this; although some OO libraries have been built, they have not been widely distributed.(excepting the Blue Obelisk community). The development of dictionaries is a major step forward in physical science for semantic interoperability.
  • support for datuments (combinations of data and documents). Conventionally chemical publications separate the text and data (the latter often as supporting information). In practice, however, most chemical data are lost. We believe that datuments will help to prevent this and allow archival for posterity.
  • interoperability with other languages. Most chemical datuments will contain some or all of the following (with supporting technology indicated):
    • text (XHTML)
    • maths (MathML)
    • graphics (SVG, PNG)
    • chemistry (CML, AnIML, ThermoML)
  • computational chemistry. There are now several programs which write CML, especially in ab initio and solid state domains. This allows users of the programs to analyse and display their results with much greater ease than before. It also makes it relatively easy to chain programs so that workflows can be built for automatic processing.
  • learning objects. CML is designed to allow annotation suitable for learning activities and to provide a wide range of different displays.
  • knowledge aggregation. Because CML uses a consistent set of markup it is possible to aggregate datuments and extract common concepts. Thus a series of journal articles in CML could be searched to extract all the melting points, spectra, reactions, etc
  • repurposing and styling. Because XML emphasizes the separation of content and style, different stylesheets can be applied to the same datument. This means that, say, an article can be written independently of a target journal and can be restyled to fit the requirements. We already work with publishers who see the CML datument as part of the primary representation of the scientific article

There is an increasing community of use in CML and this blog will address some of the common hopes and concerns. Some of these are:

  • what does CML cover and not cover?
  • what would I gain from using CML? Is it worth the effort?
  • are there sufficient CML-aware tools to do what I need?
  • do any commercial organisations use or support CML?
  • how do I start using CML?

Iâ€™ll answer the first in the next postâ€¦
